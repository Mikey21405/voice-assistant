# 为什么选择websocket而不是http?

语音对话场景需要低延迟的双向实时通信，而 HTTP 本质上是“请求-响应”模型，必须由客户端先发起请求，服务端才能返回数据；SSE 虽然支持服务端推送，但只能单向通信。相比之下，WebSocket 采用长连接机制，支持客户端与服务端之间的**全双工（通信双方可同时双向发送数据）**实时通信，因此更适合语音助手、WebRTC 等实时交互场景。

# Command信令处理模块

- 使用**策略模式+工厂模式**：对于不同信令类型，设计不同的信令处理器，通过实现ComandHandler接口来实现并注册为Bean，通过CommandDispatcher工厂调用对口的信令处理器
- BaseCommandHandler抽象基类，实现公共方法，比如发送message给对应的会话；表现出java的多态

# Agent工具调用

核心流程：构建ToolDef（工具列表）–> 调用LLM规定不是必须要返回工具，根据用户需求来选择工具，返回工具选择 –> 解析返回message，获取tool_calls解析工具列表 –> 执行对应的工具 –> 将工具调用结果以 **role: "tool"** 消息追加到对话上下文 –> 再次调用LLM生成最终回答

使用的是OpenAI兼容的Function Calling协议，用Function Calling让LLM能主动调用外部系统功能工具，而不是在prompt中硬编码；且每次请求都要带上tools的定义，因为大模型不会记住你的工具有哪些，所以继续带上工具定义可以让ai知道这个结果是哪个工具返回的，这个工具结构是怎样的，提高回答的准确率

# LLM大模型层

- 普通同步调用：使用RestTemplate.exchange发送POST请求，exchange方法是同步阻塞的，所以当请求发出后，当前线程会等待，直到LLM生成完毕返回response后才会有反应，用户需要等待，体验较差
- 流式调用：使用原生 **HttpURLConnection + SSE解析**,Restemplate默认会等待整个响应结束才会返回，不符合流式的要求，所以用SSE，SSE就是服务器会持续推送文本流，后端不断的读取行然后解析，每收到返回的token，就立刻**onToken.accept()**，直到遇到**[DONE]**，就自动结束
- ToolCalling：请求体中携带tools和tool_choice:"auto"，告诉模型有哪些工具可以用，让模型自行决定如何调用

### 为什么自己用HttpURLConnection而不是RestTemplate

因为RestTemplate是要等整个响应体生成完毕后才会返回，而SSE需要边接受边处理，是一个长期推送，
